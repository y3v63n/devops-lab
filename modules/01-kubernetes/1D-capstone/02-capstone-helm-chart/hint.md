## Task 2 — Chart.yaml with dependencies

**Hint 1:** The `dependencies` list in `Chart.yaml` must use `apiVersion: v2`. Each dependency entry needs `name`, `version` (semver range), `repository`, and optionally `condition` (a dot-notation path into `values.yaml` that enables/disables the sub-chart).

**Hint 2:** Use `x` wildcards to avoid pinning to exact patch versions:
```yaml
version: "25.x.x"
```

**Hint 3 (full Chart.yaml):**
```yaml
apiVersion: v2
name: blockchain-monitor
description: >
  Full observability stack for blockchain validator nodes.
type: application
version: 0.1.0
appVersion: "1.0.0"

dependencies:
  - name: prometheus
    version: "25.x.x"
    repository: https://prometheus-community.github.io/helm-charts
    condition: prometheus.enabled

  - name: grafana
    version: "7.x.x"
    repository: https://grafana.github.io/helm-charts
    condition: grafana.enabled
```

After saving, fetch dependencies:
```bash
helm repo add prometheus-community https://prometheus-community.github.io/helm-charts
helm repo add grafana https://grafana.github.io/helm-charts
helm repo update
cd /tmp/devops-lab/1D-02/blockchain-monitor && helm dependency update
```

## Task 3 — values.yaml

**Hint 1:** Sub-chart values are namespaced under the sub-chart name. Anything under `prometheus:` overrides the Prometheus chart's `values.yaml`; anything under `grafana:` overrides Grafana's.

**Hint 2:** The `condition` key in Chart.yaml (e.g. `condition: prometheus.enabled`) means Helm will look up `prometheus.enabled` in your values and skip the sub-chart if it is `false`.

**Hint 3 (minimal values.yaml):**
```yaml
replicaCount: 1

image:
  repository: nginx
  tag: alpine
  pullPolicy: IfNotPresent

service:
  type: ClusterIP
  port: 8080

ingress:
  enabled: true
  className: nginx
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /$2
  hosts:
    - host: ""
      paths:
        - path: /grafana(/|$)(.*)
          pathType: ImplementationSpecific
  tls: []

prometheus:
  enabled: true
  server:
    retention: 2h
    global:
      scrape_interval: 15s
  alertmanager:
    enabled: false
  prometheus-pushgateway:
    enabled: false

grafana:
  enabled: true
  adminPassword: devops-lab
  persistence:
    enabled: false
  service:
    type: ClusterIP
```

## Task 4 — Templates

**Hint 1:** Remove the boilerplate templates generated by `helm create` — they are for a generic web app, not a monitoring stack:
```bash
cd /tmp/devops-lab/1D-02/blockchain-monitor
rm -f templates/deployment.yaml templates/service.yaml \
      templates/hpa.yaml templates/serviceaccount.yaml \
      templates/ingress.yaml
```

Keep `templates/_helpers.tpl` — it defines `blockchain-monitor.fullname`, `blockchain-monitor.labels`, and `blockchain-monitor.selectorLabels` which your new templates use.

**Hint 2:** The `{{ include "blockchain-monitor.fullname" . }}` call generates a name like `release-name-blockchain-monitor`. Using this function means the release name is always embedded in resource names, preventing collisions when multiple releases are installed in the same namespace.

**Hint 3:** For the ConfigMap + Deployment + Service in one file, separate each resource with `---`. Helm renders the whole file and splits on `---` when applying.

**Hint 4 (minimal health-checker.yaml):**
```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: {{ include "blockchain-monitor.fullname" . }}-config
  labels:
    {{- include "blockchain-monitor.labels" . | nindent 4 }}
data:
  metrics: |
    # HELP node_health_status 1 if node is healthy, 0 if not
    # TYPE node_health_status gauge
    node_health_status{node="validator-1"} 1
  nginx.conf: |
    events {}
    http {
      server {
        listen 8080;
        location /metrics {
          default_type text/plain;
          alias /etc/nginx/metrics;
        }
      }
    }
---
apiVersion: apps/v1
kind: Deployment
metadata:
  name: {{ include "blockchain-monitor.fullname" . }}-health-checker
  labels:
    {{- include "blockchain-monitor.labels" . | nindent 4 }}
    app.kubernetes.io/component: health-checker
spec:
  replicas: {{ .Values.replicaCount }}
  selector:
    matchLabels:
      {{- include "blockchain-monitor.selectorLabels" . | nindent 6 }}
      app.kubernetes.io/component: health-checker
  template:
    metadata:
      labels:
        {{- include "blockchain-monitor.selectorLabels" . | nindent 8 }}
        app.kubernetes.io/component: health-checker
      annotations:
        prometheus.io/scrape: "true"
        prometheus.io/port: "8080"
        prometheus.io/path: "/metrics"
    spec:
      containers:
        - name: health-checker
          image: "{{ .Values.image.repository }}:{{ .Values.image.tag }}"
          ports:
            - containerPort: 8080
          volumeMounts:
            - name: config
              mountPath: /etc/nginx/nginx.conf
              subPath: nginx.conf
            - name: config
              mountPath: /etc/nginx/metrics
              subPath: metrics
      volumes:
        - name: config
          configMap:
            name: {{ include "blockchain-monitor.fullname" . }}-config
---
apiVersion: v1
kind: Service
metadata:
  name: {{ include "blockchain-monitor.fullname" . }}-health-checker
spec:
  type: {{ .Values.service.type }}
  ports:
    - port: {{ .Values.service.port }}
      targetPort: 8080
  selector:
    {{- include "blockchain-monitor.selectorLabels" . | nindent 4 }}
    app.kubernetes.io/component: health-checker
```

## Task 5 — Lint and render

**Hint 1:** `helm lint` catches YAML errors, missing required fields, and deprecated API versions. Always run it before installing.
```bash
helm lint /tmp/devops-lab/1D-02/blockchain-monitor/
```

**Hint 2:** `helm template` renders manifests locally without installing anything — great for inspecting what would be applied:
```bash
helm template blockchain-monitor /tmp/devops-lab/1D-02/blockchain-monitor/ \
  > /tmp/devops-lab/1D-02/rendered.yaml

# Count resources by kind
grep "^kind:" /tmp/devops-lab/1D-02/rendered.yaml | sort | uniq -c
```

**Hint 3:** Common lint errors:
- `Error: chart metadata is missing "home"` — not an error, just a warning; can be ignored.
- `[ERROR] templates/: ...` — actual errors that prevent rendering. Check the line number in the message.
- Missing `charts/` directory — run `helm dependency update` first.

## Installing the chart

Once lint passes and rendered.yaml looks correct:
```bash
kubectl create namespace monitoring
helm install blockchain-monitor /tmp/devops-lab/1D-02/blockchain-monitor/ \
  -n monitoring
```

Watch rollout:
```bash
kubectl get pods -n monitoring -w
```

Port-forward Grafana:
```bash
kubectl port-forward -n monitoring \
  svc/blockchain-monitor-grafana 3000:80 &
```

## Troubleshooting

- **"Error: found in Chart.yaml, but missing in charts/ directory"** — run `helm dependency update` inside the chart directory.
- **Sub-chart values not applied** — double-check the key nesting in values.yaml matches the sub-chart name exactly (`prometheus:`, `grafana:`).
- **Ingress 404** — make sure `minikube addons enable ingress` is active. Check `kubectl get ingress -n monitoring`.
- **Template rendering error with `.Release.Name`** — make sure you are using `{{ .Release.Name }}` (dot-delimited, inside `{{ }}`) not curly brace interpolation used in shell scripts.
